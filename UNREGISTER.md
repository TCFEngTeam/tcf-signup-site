# Cancel registration (`/unregister`)

## User flow

1. User opens `/unregister` (optional query: `?program=mhfa` or `?program=mhfa&eventId={trainingId}` to pre-select program/session).
2. User selects program, enters email, clicks **Check for Registrations** → `POST /api/unregister/lookup`.
3. If active registrations exist, the email field is replaced by a **session** dropdown. User picks a session and clicks **Email me a confirmation link** → `POST /api/unregister/request`.
4. Email contains a signed link: `/unregister/confirm?token=...`.
5. User opens the link and clicks **Yes, cancel my registration** → `POST /api/unregister/confirm`.
6. HubSpot removes the **registrant** association (or relabels — see below).

**Important:** Cancellation does **not** run when the email link is opened. Email scanners cannot cancel a registration by prefetching the page.

### What counts as “active” for unregister

- Contact must have a **registrant** association to the training (not only `unregistered`).
- Training **end date** must not have passed (uses day 2 end if present, else day 1 end). Past sessions are hidden from lookup and blocked on confirm.

### Re-registration after cancel

If `unregister.mode` is `relabel`, the contact gets an `unregistered` (or configured) label. Signing up again removes that label and creates a fresh `registrant` association.

## Configuration

### Secrets & deployment (`.env.local` / Vercel)

| Variable | Purpose |
|----------|---------|
| `UNREGISTER_TOKEN_SECRET` | HMAC secret for confirmation links (long random string) |
| `NEXT_PUBLIC_APP_URL` | Base URL in emails (e.g. `https://signup.example.com`) |
| `RESEND_API_KEY` / `RESEND_FROM_EMAIL` | Transactional email (in dev without these, the link is logged to the server console) |

### Portal wiring (`config/hubspot.json`)

| Key | Purpose |
|-----|---------|
| `associations.registrant` | Label for active registration (e.g. `registrant`) |
| `associations.cancelled` | Label after cancel when using relabel (e.g. `unregistered`) |
| `associations.registrantTypeId` / `cancelledTypeId` | Optional — HubSpot v4 numeric type IDs |
| `unregister.mode` | `remove` or `relabel` |

Token **expiry** is the training session end datetime (not a fixed hour TTL). Fallback TTL applies only when no end date exists.

---

## HubSpot modes (`unregister.mode`)

- **`remove`** — Archives only the `registrant` association. Frees capacity; no audit association.
- **`relabel`** — Archives `registrant`, then creates an association with `associations.cancelled` for reporting.

The app reads associations via HubSpot **v4** (preferred) or **v3**, and archives using the association type returned by HubSpot.

---

## HubSpot setup for `relabel`

1. Settings → Data Management → Objects → Training → Associations → Contact.
2. Add a label (e.g. `unregistered`).
3. Set `associations.cancelled` in `config/hubspot.json` to that internal name.
4. Set `unregister.mode` to `relabel`.

## Links to use in marketing / signup emails

Signup confirmation emails generated by the app already include a **direct** `/unregister/confirm?token=...` link (no need to send users to the lookup form first).

For general “cancel your registration” pages:

- MHFA: `https://your-signup-app/unregister?program=mhfa`
- QPR: `https://your-signup-app/unregister?program=qpr`
- Pre-filled session (after lookup): `.../unregister?program=mhfa&eventId={trainingRecordId}`

## API routes

| Route | Purpose |
|-------|---------|
| `POST /api/unregister/lookup` | Find active registrations for email + program |
| `POST /api/unregister/request` | Send confirmation email (requires `trainingId`) |
| `POST /api/unregister/confirm` | Perform cancellation (requires `token`) |

Rate limits apply per IP + email on lookup and request.
